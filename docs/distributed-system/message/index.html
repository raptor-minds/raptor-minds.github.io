Message Middleware

Distributed system design need to take the following key points into consideration:

  1. Reliability
  2. Scability
  3. High performance
  4. CAP (Consistency, Availablity, Partition tolerance) property
    1. The system continues to operate despite arbitrary message loss or failure of part of the system.

Kafka#

When you want to design a new distributed system, the first image in your mind should be the AKF classic model. Generally, there are three dimensions, named X,Y and Z.

The following questions we need to solve:

  • For the X axis, if we want to extend to add more machine, how to make minimum impact to the existed cluster?
    • the broker? add more?
    • for the same partition, how to scale? if the storage is not enough.
  • For the Y axis, if we want to enrich our business , which means to analyze or produce more data.
    • to add more partition?
  • For the Z axis, if we want to enlarge our cluster, we want to extend our business to the whole world, say, China, Japan, America, if we add more cluster , how can keep the CAP property.
  • How does Kafka handle the reliability problem?
    • for the broker, use HA
      • backup machine, if one crash, the others can still continue work well.
      • for the master slave work model, how to handle the election problem when the master crash
    • for the message producer:
      • we should confirm each message has alreay stored in the broker, or in the partition
    • for the message consumer:
      • we should report the the broker, where we already consumed.
      • if during the period, the broker crash, which broker else should we choose.
      • if the consumers are far less slower the producers, we should add more consumer.
        • Which partition should the new consumer handle
  • How does Kafka handle the scability?
    • use topic and partition
  • how does Kafka hanle high performance?
    • use the big memory, and the LRU algorithm to swap the space in the disk
    • keep the hot message for one period of time in the memory, the consumer can fetch or query in a high speed.

Produce and persistence message#

when message produced, how the broker tell the producer that message already received, say ACK? how the broker store the message and persistence it. To make sure high throughput under the high concurrency, if each need one ACK, is it efficienient?

  • It should be backed up, which means even the master, for the kafka it is called controller, crashed, when ACK reply to the producer, the message could be use.
  • it should be efficient, when multiple producers connect the single broker, it can reply ACK as quickly as possible.

Here comes the question:

  • when to reply the ACK message to the producer, as soon as the message received, or the message has been stored to the disk to the signle node, or the message alreay been backed up to the slave node?
  • how to reply to the producer, reactive mode, sync mode, or async mode?
  • Besides storing the message to the disk, we also need to maintain the index for the message stored on the disk, because each message's length is not the same, we can not use the array to hold the data, we need the list data structure to hold the data, so we need to introduce the index. the consumer can query the message by topic or by time, even in the sql model.

The Kafka also maintains the index for the messages and the timestamp index for the index, which can be viewed in the picture above.

Assuming that, every message should be persistenced in the disk, and each message should be consumed, does the broker need to fetch every message from the disk and send it back to the consumer?

  • â€‹ the answer is no, we can use zero copy, use sendfile(in,offset,out) to do that.

There're three kinds of ACKs in Kafka, namly 0, 1, -1 respectivly.

  • 0 stands for the ack when the message received by the broker and stored in the pagecache.
  • 1 stands for the ack when the message stored on the disk by the broker.
  • -1 stands for the ack when the message stored in the distributed system by the broker.

For the -1, there comes another problem, what kind of depth should the distributed system align with each other?

  • Half of them voted, every message should wait for half of the nodes in the cluster, would it be so long?
  • Another question, without the others slaves node backup, can these message be consumed?
    • the answer is no, here we introduce another concept.

Distribute system consistency for message backup. When should the High watermark move forward?

  • Half of them backup? The traditional distribute system solution.
  • One backup is enough?
  • When should the backup broker do this action?
    • the producer produce the message, and the ACK is -1, the producer will wait for the rest broker to finish syncing. which is called Strong consistency
    • the producer only send to the master, the follower fetch the new data from the master, Final consistency.
    • ISR (In sync replication) still keep heart beat message with each other, or with the zookeeper ?
    • OSR (out of sync replication), lose the heart beat
    • AR (Asigned replication)
    • AR = ISR + OSR

The consumer only can consume the message between Low and High watermarker.

There is no Read/Write splitting in Kafka.

Transaction#

The processing flow of RocketMQ in the distributed system can be generally descriped as followesï¼š

Two phase tx commit#

  1. Producer need to invoke A,B,C,D method in one Tx, the broker of rocket message queue will store these four message in one Half Queue.

  2. The broker will start one thread to schedully call back the execution status of A, B,C, D. When the status is OK, then the broker will put the A,B,C,D message in the Final Queue.
